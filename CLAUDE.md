## Commands

```bash
npm install       # install dependencies
npm run dev       # start Vite dev server
npm run build     # production build to dist/ (also used by CI/deploy)
```

There is no test suite and no linter configured in this repo.

## Architecture

Hugo's Kitchen Notebook is a 100% static recipe app: Vite + React + Tailwind, no backend, no build-time API calls, no database. Recipes are just Markdown files, and use metric units only (g/kg/ml/l/cm/Â°C plus tsp/tbsp) by convention.

**Recipe pipeline** (the core of the app):
1. Recipes live as `.md` files in [`_recipes/`](_recipes/) with YAML-ish frontmatter (title, prepTime, portions, tags, description, plus optional `title_nl`/`title_fr`/`title_de` and `description_nl`/`description_fr`/`description_de`) followed by a base `## Ingredients` + `## Preparation`, and optional per-language `## Ingredients (nl)` / `## Preparation (fr)` / etc. sections.
2. [`src/lib/frontmatter.js`](src/lib/frontmatter.js) hand-rolls a minimal frontmatter parser (strings/numbers/booleans/flow-lists/block-lists) â€” deliberately not a full YAML parser, to avoid Node/Buffer polyfills in the browser bundle. It parses arbitrary flat keys generically, so the `_nl`/`_fr`/`_de` suffixed keys needed nothing added here.
3. [`src/lib/recipes.js`](src/lib/recipes.js) uses `import.meta.glob('/_recipes/*.md', { query: '?raw', import: 'default', eager: true })` so Vite statically inlines every recipe file at build time â€” this is what makes the whole app buildable as a static site with zero server code. Adding a recipe is just dropping a new `.md` file in `_recipes/`; there is no registry to update. `parseSections` groups body sections by an optional `(xx)` language suffix on the `## Heading` line (e.g. `## Ingredients (nl)` â†’ name `ingredients`, lang `nl`); `loadRecipes` builds each recipe's base fields plus a `translations` map (`{ nl?, fr?, de? }`, only populated when that language actually has content). `localizeRecipe(recipe, locale)` returns the recipe with title/description/ingredients/steps swapped for a locale's translation, falling back to the base fields when that locale/recipe has none â€” components call this at render time, keyed off the current UI locale. `slugify`/routing always uses the base title, never a translation.
4. [`src/lib/scaleText.js`](src/lib/scaleText.js) detects quantities in ingredient text (mixed numbers, fractions, unicode vulgar fractions, decimals, ranges like "2-3") and scales them by a portion ratio, formatting back to the nearest "nice" fraction glyph where possible â€” locale-independent, since numbers/ranges are written identically across languages by convention (only the surrounding words change).
5. [`src/lib/safetyFlags.js`](src/lib/safetyFlags.js) flags specific ingredients (aubergine, mushroom, ricotta, feta, olive) by regex so they're visually highlighted â€” matches on whatever language is currently displayed, so coverage is incomplete for translated ingredient text beyond words that happen to be shared loanwords across en/nl/fr/de; this list is meant to be edited per-user preference, not treated as fixed logic.

**Routing**: `HashRouter` (see [`src/App.jsx`](src/App.jsx)) is used deliberately â€” `/#/recipe/...` URLs work on GitHub Pages without any 404-redirect workaround.

**Dev-only LLM recipe editor**: the chat-with-local-LLM UI lives in [`src/components/EditorPanel.jsx`](src/components/EditorPanel.jsx) (recipe list + chat + raw draft/save, extracted so one implementation can be mounted two ways) â€” either full-page at `/editor` ([`src/pages/Editor.jsx`](src/pages/Editor.jsx), just the page header + `<EditorPanel />`), or as a floating overlay available on every page via [`src/components/ChatDrawer.jsx`](src/components/ChatDrawer.jsx) (`<EditorPanel compact />`, mounted once in `App.jsx` sibling to `<Routes>` so it persists across navigation â€” lets you keep looking at a recipe while chatting instead of navigating away). Both the `/editor` route and `ChatDrawer` are registered behind `import.meta.env.DEV` so Rollup tree-shakes all of it out of the production build entirely â€” verified by grepping `dist/` for editor-only strings after a build. The LLM (Ollama, LM Studio, or any OpenAI-compatible `/chat/completions` server â€” configured via `llm.config.json`, gitignored, see `llm.config.example.json`) is reached through [`vite-plugins/recipeEditorApi.js`](vite-plugins/recipeEditorApi.js), a Vite plugin registered only via `configureServer` (dev-server-only hook, never runs during `vite build` or `vite preview`) that exposes `/api/recipes` (GET/POST/DELETE, reads/writes `_recipes/*.md` directly) and `/api/llm/chat`.

The chat proxy does real OpenAI-style tool-calling, not just text passthrough, through a small registry (`TOOL_REGISTRY` in the plugin) mapping a tool name to `{ schema, execute(args) }` â€” `execute` always returns `{ ok, resultText, extra? }`. The client's request body includes a `tools: ['save_recipe']`-style array of names it wants enabled for that conversation; the plugin looks up the matching schemas, and when the model calls one, dispatches to `execute`, feeds `resultText` back as a `role: 'tool'` message, and does a follow-up completion to get the model's natural-language confirmation before replying (any `extra` fields â€” e.g. `{ saved: {...} }` â€” are merged into the JSON response). This is what lets the user just say "save it" mid-conversation instead of manually clicking "Insert reply into draft" â†’ "Save recipe" (that manual two-button path still exists as a fallback). Failed tool calls loop back to the model for up to 3 attempts before giving up. Filename resolution (`resolveFilename` in the plugin) mirrors the client-side `deriveFilename` in `EditorPanel.jsx` â€” both slugify the frontmatter `title:` line the same way, so a chat-saved and button-saved recipe end up named identically.

Three tools are registered: `save_recipe` (writes a recipe file, used by `EditorPanel.jsx`), `update_fridge_items` (overwrites `fridge.json`'s item list, used by `Fridge.jsx`), and `update_recipe_tags` (rewrites just the `tags:` frontmatter line of one existing recipe, used by the "Auto-tag all recipes" button â€” `EditorPanel.jsx`'s `autoTagAll` loops over every recipe sequentially, one `/api/llm/chat` call each with [`src/lib/autoTagPrompt.js`](src/lib/autoTagPrompt.js) as the system prompt and that recipe's full raw content as context). Adding a fourth tool means: define its schema + `execute` in `TOOL_REGISTRY`, and pass its name in the client's `tools` array.

The exact markdown shape the LLM must produce â€” now including the `_nl`/`_fr`/`_de` frontmatter keys and the eight `##` sections (base + 3 translations, for both Ingredients and Preparation) â€” is defined in [`src/lib/recipeFormatPrompt.js`](src/lib/recipeFormatPrompt.js); keep it in sync with `frontmatter.js`/`recipes.js` parsing rules if those change.

Hugo's standing personal preferences (portion-size defaults, step-detail expectations) are **live-editable, not just a source file**: `GET`/`POST /api/preferences` (same plugin) read/write `preferences.json` (gitignored), seeded from `DEFAULT_PREFERENCES_TEXT` in the plugin the first time it's requested. [`src/lib/recipePreferences.js`](src/lib/recipePreferences.js)'s `RECIPE_PREFERENCES` export is now only the client's initial-render/offline fallback (shown before the `/api/preferences` fetch resolves) â€” keep its text in sync with the plugin's `DEFAULT_PREFERENCES_TEXT` seed, since they're meant to match. `EditorPanel.jsx` fetches the live text on mount into `preferencesText`, concatenates it with `RECIPE_FORMAT_SYSTEM_PROMPT` via a `useMemo`'d `systemPrompt`, and exposes a gear-icon modal (`showPreferences`/`preferencesDraft`) to view/edit/save it â€” this is "memory management" from the UI, no code editing required for a preference change.

`llm.config.json` has two fields specifically for hybrid-reasoning models (Qwen3.5-style, i.e. models that think-by-default): `reasoningEffort` (sent as `reasoning_effort` on the OpenAI-compat path) and `nativeOllama` (bool). When `nativeOllama: true`, `callLlm` in the plugin targets Ollama's **native** `/api/chat` instead of `${baseUrl}/chat/completions`, sending `think: false` instead of `reasoning_effort` â€” this exists because Ollama's OpenAI-compat endpoint silently drops both params once `tools` are present (every request here sends tools), causing multi-minute hangs or empty `content` on a reasoning model. Native mode also returns `tool_calls[].function.arguments` pre-parsed as an object rather than a JSON string; `parseToolArguments()` normalizes both shapes â€” keep it even if you revert to a non-reasoning model, since it's a harmless no-op there.

The recipe draft panel renders a **live preview** by default, not raw markdown â€” `parseRecipeMarkdown(draft)` (exported from `recipes.js`, factored out of `loadRecipes` so it can run over in-progress textarea content, not just files on disk) parses the current draft into the same shape as a real recipe, `localizeRecipe` swaps in whichever of its `translations` the user picks via language tabs (only shown when the draft actually has more than the base language), and `RecipePreview` renders it exactly like the real recipe page (title/description/tags/ingredients/steps) instead of showing `---`/`##` markdown soup. An "Edit raw markdown" toggle still exposes the underlying textarea for manual edits/paste â€” that's the only place raw markdown is intentionally shown.

`validateRecipeMarkdown` in the plugin runs before every write (manual POST and tool-called save alike) and exists because small local models occasionally emit content that looks right but silently breaks parsing â€” seen in practice: a missing closing `---` (frontmatter.js then treats the entire file as unparsed body, so title/tags/portions vanish with no error) and `tag:` instead of `tags:`. Those are hard `errors` that block the write. Missing NL/FR/DE frontmatter keys or sections are `warnings` instead â€” they never block a save (the app just falls back to the base language for that field), but on the tool-call path they're appended to the "Saved as X.md" tool-result message so the model can see what's missing and offer to fill it in. A hard-error failure is fed back to the model as a `role: 'tool'` error message so it can retry, capped at 3 attempts before falling back to a message pointing the user at the manual draft box.

**Fridge**: [`src/pages/Fridge.jsx`](src/pages/Fridge.jsx) (route `/fridge`, dev-only same as `/editor`) is a small pantry tracker â€” a chat (using the `update_fridge_items` tool and [`src/lib/fridgeAssistantPrompt.js`](src/lib/fridgeAssistantPrompt.js) as its system prompt) lets the user say what they have/used, in any of the four languages, and the model normalizes it to short English ingredient names in `fridge.json` (gitignored, read/written via `/api/fridge` GET/POST in the same plugin). The "What can I cook?" button doesn't use a tool at all â€” it just sends a user message with the current fridge items plus a compact `title: ingredients` summary of every recipe (built client-side from the already-imported `recipes` array) and asks the model to rank matches conversationally; matching is base-English-only (doesn't consider `translations`), which is fine since it's just for the model's own reasoning, not rendered.

**i18n**: both UI chrome and recipe content (title/description/ingredients/steps, when a recipe provides that language â€” see the recipe pipeline section above) are translatable across English/Dutch/French/German via a small hand-rolled layer â€” no i18next, consistent with this repo's general avoidance of dependencies for small surface areas. [`src/lib/translations.js`](src/lib/translations.js) holds the UI-chrome dictionaries + `translate(locale, key, params)` (supports `{placeholder}` interpolation and a `_one`/`_other` plural suffix keyed off `params.count`); recipe-content translation is separate, handled by `localizeRecipe` in `recipes.js`. [`src/hooks/useLocale.jsx`](src/hooks/useLocale.jsx) provides `LocaleProvider`/`useTranslation()` via React Context â€” Context (not a per-component hook like `useDarkMode`) is required here because changing the language must re-render every consumer at once, not just the component that owns the toggle. [`src/components/LanguageSwitcher.jsx`](src/components/LanguageSwitcher.jsx) is the `<select>` dropdown; language names are always shown untranslated in their own language (English/Nederlands/FranĂ§ais/Deutsch) so a user can find their language regardless of the current locale. Tags are never translated â€” they're a single shared list of English kebab-case identifiers used for filtering, in every language.

**Deployment**: [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml) builds with `npm ci && npm run build` and publishes `dist/` via `actions/upload-pages-artifact` + `actions/deploy-pages`. Two things must be kept in sync for Pages to work:
- `REPO_NAME` (the Vite `base` path) in [`vite.config.js`](vite.config.js) must match the actual GitHub repo name (e.g. `/recipe-app/`), or `/` for a `<user>.github.io` repo.
- The repo's **Settings â†’ Pages â†’ Build and deployment â†’ Source** must be set to **GitHub Actions** (this is a one-time manual step in the GitHub UI, not something the workflow file can set).
